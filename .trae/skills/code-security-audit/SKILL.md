# 需求验证 Skill

## 核心定位

本 Skill 是一个**验收框架**，提供 L1-L4 四层证据收集模板，不预设具体实现细节，让 AI 发挥主观能动性去验证需求实现是否正确。

## 工作模式

```
用户输入需求 → AI 分析需求 → 搜索相关代码 → L1-L4 证据收集 → 输出验收报告
```

## 输入格式

```
需求：[需求描述]
```

示例：
- "需求：新增用户注册功能，包含邮箱验证和密码强度检查"
- "需求：修复订单查询接口的性能问题"
- "需求：实现 JWT 认证中间件"

---

# L1-L4 证据收集框架

## L1 - 需求理解与范围界定

**目标**：确认需求的理解正确性和实现范围

| 检查项 | 说明 | 证据形式 |
|--------|------|---------|
| 需求拆解 | 需求包含哪些核心功能点 | 功能点清单 |
| 边界定义 | 需求的输入输出边界 | 输入输出规格 |
| 依赖识别 | 需要哪些依赖模块 | 依赖模块清单 |
| 验收标准 | 如何判断需求已完成 | 验收条件 |

**自问自答示例**：
- "需求的核心功能点是什么？"
- "需求的输入输出是什么？"
- "需要依赖哪些现有模块？"

---

## L2 - 实现存在性验证

**目标**：确认需求已被实现，代码存在

| 检查项 | 说明 | 证据形式 |
|--------|------|---------|
| 代码存在 | 核心代码文件是否存在 | 文件路径 |
| 功能实现 | 核心功能是否已编码 | 代码片段 |
| 接口定义 | 公开接口是否定义 | 接口签名 |
| 依赖注入 | 依赖是否正确注入 | DI 注册代码 |

**自问自答示例**：
- "实现需求的代码文件在哪里？"
- "核心功能的代码片段是什么？"
- "接口定义是否完整？"

---

## L3 - 实现正确性验证

**目标**：确认实现逻辑正确，符合需求

| 检查项 | 说明 | 证据形式 |
|--------|------|---------|
| 逻辑正确性 | 业务逻辑是否正确 | 代码分析 |
| 数据流完整 | 输入→处理→输出是否完整 | 数据流分析 |
| 边界处理 | 异常情况是否处理 | 异常处理代码 |
| 调用链路 | 模块间调用是否正确 | 调用链分析 |

**自问自答示例**：
- "业务逻辑是否符合需求？"
- "数据流是否完整？"
- "异常情况是否处理？"

---

## L4 - 质量与安全性验证

**目标**：确认实现质量达标，无安全隐患

| 检查项 | 说明 | 证据形式 |
|--------|------|---------|
| 安全检查 | 是否存在安全漏洞 | 安全分析 |
| 代码质量 | 代码是否符合规范 | 代码审查 |
| 测试覆盖 | 是否有单元测试 | 测试代码 |
| 文档完整 | 是否有必要文档 | 文档链接 |

**自问自答示例**：
- "是否存在安全隐患？"
- "代码是否符合规范？"
- "是否有单元测试？"

# 行为准则

1. **逐层递进**：L1 通过后才检查 L2，依此类推
2. **代码即证据**：所有结论必须有代码片段支持
3. **主观能动性**：根据需求灵活搜索和分析，不局限于固定步骤
4. **诚实标注**：不确定的地方标注 CONDITIONAL
5. **证据持久化**：自动保存报告和证据

# 使用示例

## 示例 1：功能验收

```
需求：新增用户注册功能，包含邮箱验证和密码强度检查
```

**AI 自动执行**：
1. 分析需求：用户注册、邮箱验证、密码强度检查
2. 搜索相关代码：UserService.cs、RegisterDto.cs、AccountController.cs
3. L1：确认需求拆解正确
4. L2：确认代码存在
5. L3：验证逻辑正确性
6. L4：检查安全性和质量

## 示例 2：Bug 修复验证

```
需求：修复订单查询接口的 SQL 注入漏洞
```

**AI 自动执行**：
1. 分析需求：修复 SQL 注入漏洞
