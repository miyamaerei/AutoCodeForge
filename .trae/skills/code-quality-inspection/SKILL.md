# 代码质量检查 Skill

## 目标和范围

对 AutoCodeForge 项目的代码变更进行质量检查，包括放置正确性、复用合规性、闭环完整性。必须依赖技能 02（模块影响发现）和技能 01（系统知识）进行证据验证。

## 输入/输出契约

### 输入
- **变更文件列表**: 本次提交涉及的文件路径
- **变更类型**: `add`（新增）| `modify`（修改）| `delete`（删除）
- **模块名称**: 主要影响的模块

### 输出
固定格式的质量检查报告，包含：
1. 放置检查结果（文件位置是否符合架构）
2. 复用检查结果（是否正确复用基类和组件）
3. 闭环检查结果（前后端契约、测试覆盖）
4. 风险分级（通过/有条件通过/失败）
5. 修复措施（可操作的改进建议）

## 三个检查阶段

### 阶段 1: 放置检查（Placement Check）

#### 检查目标
验证新增/修改文件是否放置在正确的架构层和目录中。

#### 检查规则

**后端四层架构规则**:
| 文件类型 | 正确位置 | 禁止位置 | 检查方法 |
|---------|---------|---------|---------|
| 实体（Entity） | `server/src/AutoCodeForge.Core/Entities/` | 其他任何位置 | 文件路径匹配 |
| DTO | `server/src/AutoCodeForge.Core/DTOs/` | `Entities/` 或 `Services/` | 文件路径匹配 |
| 仓储接口 | `server/src/AutoCodeForge.Core/Interfaces/` | `Infrastructure/` | 文件路径匹配 |
| 仓储实现 | `server/src/AutoCodeForge.Infrastructure/Repositories/` | `Core/` 或 `Application/` | 文件路径匹配 + 继承 `BaseRepository<T>` |
| 业务服务 | `server/src/AutoCodeForge.Application/Services/` | `Core/` 或 `Api/` | 文件路径匹配 + 后缀 `Service` |
| HTTP 端点 | `server/src/AutoCodeForge.Api/Endpoints/` | `Application/` | 文件路径匹配 + 后缀 `Endpoints` |
| 中间件 | `server/src/AutoCodeForge.Api/Middleware/` | 其他位置 | 文件路径匹配 + 后缀 `Middleware` |

**前端模块结构规则**:
| 文件类型 | 正确位置 | 禁止位置 | 检查方法 |
|---------|---------|---------|---------|
| Types | `client/src/modules/[module]/[module].types.ts` | `views/` 或 `store/` | 文件路径匹配 + 后缀 `.types.ts` |
| API 调用 | `client/src/modules/[module]/[module].api.ts` | `views/` | 文件路径匹配 + 后缀 `.api.ts` |
| Store | `client/src/modules/[module]/store/` | 根目录 | 文件路径匹配 + `defineStore` |
| 路由 | `client/src/modules/[module]/routes.ts` | `views/` | 文件路径匹配 + 导出 `RouteRecordRaw[]` |
| 页面组件 | `client/src/modules/[module]/views/` | 根目录或 `components/` | 文件路径匹配 + `.vue` |
| 共享组件 | `client/src/components/` | `modules/[module]/` | 全局可复用组件 |

#### 检查流程

1. **调用技能 01** 获取架构层定义
2. **遍历变更文件列表**：
   - 检查文件路径是否符合规则
   - 检查文件命名是否符合规范
   - 检查文件内容是否与位置匹配（如：Service 类在 Services/ 目录）

3. **记录违规项**：
   - 文件路径不符合规则
   - 命名不符合规范
   - 内容与位置不匹配

**判定标准**:
- ✅ **通过**: 所有文件放置正确
- ⚠️ **警告**: 有 1-2 个轻微违规（如命名不规范）
- ❌ **失败**: 有跨层引用或严重违规（如 Core 层引用 Infrastructure）

---

### 阶段 2: 复用检查（Reuse Check）

#### 检查目标
验证代码是否正确复用 AutoCodeForge 的基类、基础组件和公共工具，避免重复实现。

#### 复用规则

**后端强制复用清单**:
| 基类/组件 | 用途 | 检查规则 | 证据文件 |
|---------|------|---------|---------|
| `BaseRepository<T>` | 仓储基类 | 所有仓储必须继承此基类 | `server/src/AutoCodeForge.Infrastructure/Repositories/Base/BaseRepository.cs` |
| `UserOwnedEntity` | 用户隔离实体 | 需要用户隔离的实体必须继承 | `server/src/AutoCodeForge.Core/Entities/Base/UserOwnedEntity.cs` |
| `AuditableEntity` | 可审计实体 | 需要审计的实体必须继承 | `server/src/AutoCodeForge.Core/Entities/Base/AuditableEntity.cs` |
| `ApiResponse<T>` | 统一响应模型 | 所有端点响应必须用此模型 | `server/src/AutoCodeForge.Core/Models/ApiResponse.cs` |
| `ExceptionHandlingMiddleware` | 全局异常处理 | 不得在端点中裸抛异常 | `server/src/AutoCodeForge.Api/Middleware/ExceptionHandlingMiddleware.cs` |
| `JwtAuthMiddleware` | JWT 认证 | 不得自己实现认证逻辑 | `server/src/AutoCodeForge.Api/Middleware/JwtAuthMiddleware.cs` |

**前端强制复用清单**:
| 组件/工具 | 用途 | 检查规则 | 证据文件 |
|---------|------|---------|---------|
| Axios 实例 | HTTP 客户端 | 必须使用配置好的 Axios 实例，不得直接用 `fetch` | `client/src/api/` |
| Pinia Setup Store | 状态管理 | 必须使用 `defineStore` setup 格式 | 所有 Store 文件 |
| Element Plus 组件 | UI 组件 | 优先使用 Element Plus，避免自己实现基础组件 | `client/package.json#L10` |
| VeeValidate + Zod | 表单验证 | 表单验证必须用此组合，不得自己实现 | `client/package.json#L15-L16` |

#### 检查流程

1. **调用技能 02** 获取模块依赖关系
2. **扫描新增代码**：
   - 后端：检查 `class XXXRepository` 是否继承 `BaseRepository<T>`
   - 后端：检查实体是否继承 `UserOwnedEntity` 或 `AuditableEntity`
   - 后端：检查端点返回是否用 `ApiResponse<T>`
   - 前端：检查是否使用了 `defineStore`
   - 前端：检查是否使用了 Axios 而非 `fetch`

3. **识别违规模式**：
   - 🔴 **严重违规**: 重新实现了基类功能（如自己写数据库访问）
   - 🟡 **中度违规**: 未继承基类但功能正常
   - 🟢 **轻微违规**: 命名不规范但逻辑正确

**判定标准**:
- ✅ **通过**: 正确复用所有基类和组件
- ⚠️ **有条件通过**: 有 1-2 个中度违规，需记录技术债
- ❌ **失败**: 有严重违规，必须重构

---

### 阶段 3: 闭环检查（Closure Check）

#### 检查目标
验证前后端契约一致、测试覆盖完整、文档同步更新，确保变更形成闭环。

#### 闭环检查项

**前后端契约检查**:
| 检查项 | 后端证据 | 前端证据 | 检查方法 |
|-------|---------|---------|---------|
| DTO 字段一致 | `[Dto].cs` 中的属性 | `[module].types.ts` 中的接口 | 逐字段对比 |
| 枚举值一致 | `[Enum].cs` 中的枚举 | `[module].types.ts` 中的枚举 | 值和名称对比 |
| API 路由一致 | `[Endpoints].cs` 中的路由 | `[module].api.ts` 中的 URL | 路由路径对比 |
| 请求参数一致 | `[Request].cs` 中的属性 | API 方法的参数类型 | 参数类型对比 |
| 响应数据一致 | `[Dto].cs` 返回类型 | API 方法的返回类型 | 返回类型对比 |

**测试覆盖检查**:
| 检查项 | 覆盖标准 | 检查方法 | 证据文件 |
|-------|---------|---------|---------|
| 后端单元测试 | 核心业务方法 ≥ 80% | 检查 `[Service]Tests.cs` 是否存在 | `server/tests/AutoCodeForge.Tests/` |
| 前端单元测试 | Store Action ≥ 80% | 检查 `[module].spec.ts` 是否存在 | `client/src/modules/[module]/__tests__/` |
| 集成测试 | 端到端流程覆盖 | 检查端到端测试用例 | 待确认测试目录 |

**文档同步检查**:
| 检查项 | 检查内容 | 证据文件 |
|-------|---------|---------|
| Swagger 文档 | 新增端点有 XML 注释 | `[Endpoints].cs` 中的 `///` 注释 |
| README 更新 | 重大功能变更需更新 README | `README.md` |
| API 文档 | API 契约变更需更新文档 | `docs/` 中的 API 文档 |

#### 检查流程

1. **前后端契约检查**：
   - 读取后端 DTO 文件，提取字段列表
   - 读取前端 Types 文件，提取接口定义
   - 逐字段对比，记录不一致项

2. **测试覆盖检查**：
   - 调用技能 02 获取模块文件列表
   - 检查对应测试文件是否存在
   - 检查测试用例覆盖核心方法

3. **文档同步检查**：
   - 检查新增端点是否有 XML 注释
   - 检查 README 是否提及新功能（如为重大功能）

**判定标准**:
- ✅ **通过**: 契约一致、测试覆盖 ≥ 80%、文档已更新
- ⚠️ **有条件通过**: 测试覆盖 60-80%，文档待补充
- ❌ **失败**: 契约不一致或测试覆盖 < 60%

## 依赖规则

本技能必须依赖以下技能：

1. **技能 02（模块影响发现）**：
   - 用途: 获取模块文件列表、依赖关系
   - 调用时机: 复用检查、闭环检查

2. **技能 01（系统知识）**：
   - 用途: 获取架构层定义、技术栈规范
   - 调用时机: 放置检查、复用检查

**调用示例**:
```
1. 调用技能 01 获取架构层定义
2. 调用技能 02 获取模块 [task-center] 的文件列表
3. 执行放置检查：验证文件路径符合架构
4. 执行复用检查：验证基类继承和组件复用
5. 执行闭环检查：验证契约一致和测试覆盖
6. 生成质量检查报告
```

## 风险分级模型

### 风险评分规则

| 检查项 | 权重 | 评分规则 |
|-------|------|---------|
| 放置检查 | 30% | 每个违规 -10 分 |
| 复用检查 | 40% | 严重违规 -20 分，中度违规 -10 分，轻微违规 -5 分 |
| 闭环检查 | 30% | 契约不一致 -15 分，测试覆盖不足 -10 分，文档缺失 -5 分 |

**总分**: 100 分（满分）

### 风险等级划分

| 总分范围 | 风险等级 | 判定结果 | 处理措施 |
|---------|---------|---------|---------|
| 90-100 | 🟢 低风险 | ✅ 通过 | 可直接合并 |
| 70-89 | 🟡 中风险 | ⚠️ 有条件通过 | 记录技术债，下一迭代修复 |
| 0-69 | 🔴 高风险 | ❌ 失败 | 必须修复后重新提交 |
