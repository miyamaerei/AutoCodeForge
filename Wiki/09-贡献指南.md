# 贡献指南

欢迎参与 AutoCodeForge 项目！本指南将帮助您了解如何贡献代码、报告问题或提出建议。

## 贡献方式

### 1. 报告 Bug

如果您发现 Bug，请通过 [GitHub Issues](https://github.com/miyamaerei/AutoCodeForge/issues) 提交。

**提交 Bug 报告时请包含**:
- 清晰的标题和描述
- 复现步骤（详细）
- 预期行为和实际行为
- 环境信息：
  - OS（Windows/Linux/macOS）
  - Node.js 版本
  - .NET 版本
  - 浏览器类型和版本（如适用）
- 错误日志或截图
- 代码示例（如适用）

**Bug 报告模板**:
```markdown
### 描述
简要描述 Bug

### 复现步骤
1. 访问 xxx 页面
2. 点击 xxx 按钮
3. 输入 xxx 数据
4. 观察到错误

### 预期行为
应该...

### 实际行为
实际...

### 环境信息
- OS: Windows 11
- Node.js: v20.19.0
- .NET: 10.0
- 浏览器: Chrome 120

### 错误日志
```
粘贴错误日志
```

### 截图
[添加截图]
```

### 3. 贡献代码

#### 开发流程

1. **Fork 仓库**:
   - 访问 https://github.com/miyamaerei/AutoCodeForge
   - 点击右上角 "Fork" 按钮

2. **克隆到本地**:
   ```bash
   git clone https://github.com/your-username/AutoCodeForge.git
   cd AutoCodeForge
   ```

3. **添加上游仓库**:
   ```bash
   git remote add upstream https://github.com/miyamaerei/AutoCodeForge.git
   ```

4. **创建功能分支**:
   ```bash
   git checkout -b feature/your-feature-name
   ```

5. **开发和测试**:
   - 遵循代码规范（见下文）
   - 编写单元测试
   - 确保所有测试通过

6. **提交代码**:
   ```bash
   git add .
   git commit -m "feat(module): 添加新功能"
   ```

7. **同步上游更新**:
   ```bash
   git fetch upstream
   git rebase upstream/main
   ```

8. **推送到 Fork**:
   ```bash
   git push origin feature/your-feature-name
   ```

9. **创建 Pull Request**:
   - 访问您的 Fork 仓库
   - 点击 "Compare & pull request"
   - 填写 PR 描述（见下文 PR 模板）
   - 提交 PR

## 代码规范

### 前端代码规范

参见 [开发指南 - 前端代码规范](05-开发指南.md#前端代码规范-typescript)

**核心规范**:
- 使用 TypeScript
- 使用 Composition API（Setup 语法）
- 使用 Pinia Setup Store
- 遵循 ESLint 规则
- 组件文件名使用 PascalCase
- 文件名使用 kebab-case
- 编写 JSDoc 注释

**运行代码检查**:
```bash
cd client

# 运行 ESLint
npm run lint

# 自动修复
npm run lint:fix

# 类型检查
npm run type-check
```

---

### 后端代码规范

参见 [开发指南 - 后端代码规范](05-开发指南.md#后端代码规范-c)

**核心规范**:
- 使用 C# 12 语言特性
- 异步方法以 `Async` 结尾
- 所有公共方法编写 XML 注释
- 仓储继承 `BaseRepository<T>`
- 服务通过接口注入
- 遵循 SOLID 原则

**运行代码检查**:
```bash
cd server

# 构建并检查警告
dotnet build

# 运行测试
dotnet test

# 代码格式化
dotnet format
```

## 提交规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范。

**提交消息格式**:
```
<类型>(<范围>): <简短描述>

[可选的详细描述]

[可选的 Footer]
```

**提交类型**:
| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 代码重构 |
| `test` | 测试相关 |
| `chore` | 构建/工具相关 |
| `perf` | 性能优化 |

**示例**:
```bash
# 简单提交
git commit -m "feat(task-center): 添加任务优先级字段"

# 详细提交
git commit -m "fix(auth): 修复登录令牌刷新逻辑

令牌过期后无法正常刷新，导致用户频繁退出登录。

修复逻辑：
- 在令牌过期前 5 分钟自动刷新
- 刷新失败时清除本地令牌并重定向到登录页

Closes #123"
```

### PR 审查流程

1. **自动检查**:
   - CI/CD 构建和测试
   - 代码质量检查（ESLint、StyleCop）
   - 测试覆盖率检查

2. **人工审查**:
   - 至少 1 名维护者审查
   - 审查要点：
     - 代码质量和可维护性
     - 性能影响
     - 安全性考虑
     - 测试覆盖率
     - 文档完整性

3. **反馈和修改**:
   - 根据审查意见修改代码
   - 回复审查评论
   - 推送更新后自动触发重新检查

4. **合并**:
   - 所有检查通过
   - 所有审查意见解决
   - 维护者合并 PR

## 社区行为准则

### 基本原则

1. **尊重**: 尊重所有贡献者，友善交流
2. **包容**: 欢迎不同背景和经验的贡献者
3. **建设性**: 提供建设性反馈，避免攻击性语言
4. **专业**: 保持专业和礼貌的沟通方式

### 不可接受的行为

- 骚扰、歧视或侮辱性言论
- 人身攻击或政治攻击
- 公开或私下骚扰
- 未经许可发布他人隐私信息
- 其他不道德或不专业的行为
