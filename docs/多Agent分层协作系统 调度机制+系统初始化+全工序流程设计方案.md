# 多Agent分层协作系统 调度机制\+系统初始化\+全工序流程设计方案

## 一、整体前置结论

1\. 你定义的 **需求、查询当前信息、计划、开发、测试、提交、审核** 7个工序，是**串行流水线\+节点可重试、可回退、可驳回**的标准研发流程，完全适配当前多秘书/老大/小弟的组织架构。

2\. 整套系统调度不依赖定时轮询，采用 **事件驱动\+状态机调度\+任务工序驱动**，解决卡顿、堆叠、无序执行问题。

3\. 初始化设计分为：**数据库初始化、Agent角色实例初始化、任务工序模板初始化、调度器初始化**四层，代码可直接落地。

4\. 所有工序进度、节点状态、流转记录**全部落库**，完全贴合上一章五张数据库表结构。

## 二、核心调度架构设计（最终落地版）

### 2\.1 调度模式：放弃轮询，采用「事件驱动\+状态触发」

初代方案：10分钟定时轮询 → 资源浪费、任务堆叠、卡顿阻塞

新版最终方案：**任务状态变更即触发下一个节点**

链路：

任务创建\(需求\) → 自动触发信息查询 → 查询完成触发计划 → 计划完成触发开发 → 开发完成触发测试 → 测试完成触发提交 → 提交完成触发审核 → 审核完成闭环

无任务不运行、有任务自动流转、卡住自动熔断重试。

### 2\.2 调度核心规则（适配你的Agent角色）

- **秘书Agent**：全局调度器、工序触发器、入口出口、超时监控，支持**多实例部署**，多秘书并行处理不同任务

- **部门老大Agent**：每一道工序的审核者、分支决策者、回退审批者

- **小弟Agent**：每一道工序的实际执行者（需求分析/查询/计划/开发/测试/提交）

**核心分工公式**：

小弟 = 做事（执行工序）

老大 = 审事（卡点审核、不通过打回上一步）

秘书 = 控事（调度流转、状态推进、超时兜底、结果归档），多秘书并行提升吞吐量

## 三、7大工序标准化拆分（完全贴合你的需求）

将你的流程固化为**系统固定流水线模板**，全局统一执行，不混乱、不跳步、不逆序。

工序顺序固定：

**1\.需求梳理 → 2\.查询当前信息 → 3\.方案计划 → 4\.代码开发 → 5\.测试校验 → 6\.版本提交 → 7\.最终审核**

### 3\.1 各工序权责分配

|工序节点|执行角色|审核角色|节点产出物|失败处理逻辑|
|---|---|---|---|---|
|1\.需求梳理|小弟|部门老大|需求说明书、任务拆解清单|打回重梳理，2次失败终止任务|
|2\.查询当前信息|小弟|部门老大|现状调研文档、环境/数据/版本信息|重新查询，补全信息|
|3\.方案计划|小弟|部门老大|执行方案、排期、技术路线|方案重构，重新规划|
|4\.代码开发|小弟|部门老大|业务代码、功能模块、注释文档|bug修复、重新开发|
|5\.测试校验|小弟|部门老大|测试报告、bug清单、通过率数据|回归测试、修复复测|
|6\.版本提交|小弟|部门老大|PR信息、提交记录、版本号|补提、重新合并|
|7\.最终审核|部门老大|秘书兜底终审|最终验收结论、完整交付物|整体打回，重启流水线|

## 四、系统完整初始化设计（代码落地核心）

系统启动执行 **一次性初始化**，分为4层，启动后直接可用，无需手动配置。

### 4\.1 第一层：数据库初始化（表\+基础数据预置）

服务首次启动自动执行：

1. 初始化5张核心表（dept\_info、agent\_info、task\_main、task\_flow\_log、task\_exception\_log）

2. 预置默认部门：研发部（统一承载全流程研发任务）

3. 预置三类默认Agent实例（常驻节点）

**初始化预置Agent数据规则**：

- 秘书Agent：预置多个实例，负责所有任务调度收口，空闲秘书竞争接单，互为容错备份

- 部门老大Agent：1个研发部负责人

- 小弟Agent：默认初始化3个，用于负载均衡执行7道工序

所有Agent初始状态 = **idle 空闲**，可直接接单。

### 4\.2 第二层：状态机初始化（全局枚举注册）

系统启动加载两套状态机，全局常驻内存：

#### 1）Agent状态机

idle(空闲) ↔ handling(处理问题) ↔ learning(学习)

#### 2）任务工序状态机

wait\_allot → need\_query → planning → developing → testing → committing → auditing → finish/fail

每一个工序对应一个独立任务状态，调度器根据状态判断下一步流转。

### 4\.3 第三层：流水线模板初始化（核心）

系统启动预加载 **固定7步流水线模板**，所有新任务默认套用该模板：

```python

# 伪代码：系统启动初始化流水线模板
PIPELINE_TEMPLATE = [
    "demand_analyse",    # 需求梳理
    "query_current",    # 查询当前信息
    "make_plan",        # 方案计划
    "development",      # 开发
    "test_verify",      # 测试
    "commit_pr",        # 提交PR
    "final_audit"       # 最终审核
]
    
```

作用：任何任务进来，自动按顺序跑完全流程，不会乱序、不会漏步骤。

### 4\.4 第四层：调度器初始化（常驻后台）

系统启动后常驻3个调度线程，全程自动跑：

1. **任务分发调度器**：监听空闲Agent、分配新工序任务

2. **工序流转调度器**：监听工序完成，自动推进下一个节点

3. **异常兜底调度器**：监听超时、失败、卡死任务，自动重试/熔断

## 五、单次任务完整初始化\+流转代码逻辑（可直接落地）

### 5\.1 新任务接入初始化流程

用户/API传入需求 → 空闲秘书Agent竞争获取 → 执行初始化：

1. 生成唯一 task\_id

2. 写入 task\_main 任务主表，状态初始化为 wait\_allot

3\. 绑定对应秘书实例、所属部门

4. 写入第一条流转日志：任务创建、等待分配

5. 调度器捕获新任务，触发第一步工序：需求梳理

### 5\.2 逐工序自动调度逻辑

**Step1：需求梳理**

调度器筛选空闲小弟 → 分配任务 → 小弟状态改为handling(处理问题) → 产出需求文档 → 提交老大审核

老大审核通过 → 调度器触发下一步【查询当前信息】

审核驳回 → 打回小弟重写，记录异常日志

**Step2\~Step6：查询/计划/开发/测试/提交**

完全复用同一套流转逻辑：小弟执行 → 老大审核 → 通过进下一工序、驳回回退

**Step7：最终审核闭环**

老大终审 → 秘书兜底校验 → 任务标记finish → 归档所有产出物与日志 → 所有Agent状态重置idle

## 六、Agent角色技能体系 \+ 专属学习机制（核心补全）

此前架构完成了**职务分工、任务调度、工序流转**，但缺失最关键的一环：**角色专属技能池、差异化学习、在岗能力迭代**。

核心问题修复：**不是所有Agent学一样的东西，而是「什么岗位、什么技能、学什么内容」**，学习完全贴合自身工作职责与工序场景。

### 6\.1 核心设计思想

- 每个Agent = **职务身份 \+ 专属技能池 \+ 独立学习库**

- 任务干活靠「技能」，任务优化靠「学习」

- 学习不是通用训练，是**岗位垂直技能迭代**，只学自己未来要用到的能力

- 学习状态（learning）不再是摆设，是**可调度、可落库、可产出能力升级**的正式环节

### 6\.2 三类Agent 1:1专属技能配置（系统初始化预置）

系统启动初始化时，不仅初始化角色，同时**自动绑定对应岗位技能池**，永久生效、可迭代扩展。

#### 6\.2\.1 秘书Agent（全局调度型技能，多实例）

**岗位定位**：调度、收口、流转、接口响应、日志归档、异常监控，多个秘书并行处理不同任务

**专属技能池（初始化写入）**

- 任务合法性校验、权限校验、参数清洗技能

- 任务优先级智能分发技能

- 全流程流转把控、节点推进调度技能

- 任务超时、异常兜底监控技能

- 结果格式化、日志规整、归档汇总技能

**秘书专属学习方向**

- 学习历史失败任务的**调度卡点**，优化分发逻辑

- 学习高频需求类型，优化任务分类、部门匹配规则

- 学习异常场景，优化兜底、预警、重试策略

- 学习多秘书协同调度策略，优化任务分配均衡性

#### 6\.2\.2 部门老大Agent（决策审核型技能）

**岗位定位**：任务拆解、方案评审、结果审核、跨部门流转、问题判定

**专属技能池（初始化写入）**

- 复杂需求拆解、子任务拆分技能

- 方案可行性评审、技术路线研判技能

- 产出结果合规性、完整性、准确性审核技能

- 问题驳回、缺陷定位、整改指导技能

- 跨部门协作、任务流转、资源调配技能

**老大专属学习方向**

- 学习历史驳回案例，沉淀**审核标准SOP**

- 学习高频任务缺陷点，提前规避工序风险

- 学习任务拆解最优范式，提升小弟执行效率

- 学习跨部门协作场景，优化流转决策

#### 6\.2\.3 小弟Agent（落地执行型技能）

**岗位定位**：全工序落地执行，覆盖7大核心流程

**专属技能池（初始化写入，完全对齐你的7道工序）**

- 需求梳理技能：需求拆解、痛点提炼、范围界定

- 信息查询技能：现状调研、环境核查、数据比对、信息汇总

- 方案计划技能：技术方案撰写、排期规划、风险预判

- 代码开发技能：业务逻辑实现、模块开发、代码规范

- 测试校验技能：用例设计、功能测试、bug定位、回归验证

- 版本提交技能：代码合并、PR提交、版本备注、冲突处理

- 交付产出技能：文档输出、结果规整、日志补齐

**小弟专属学习方向**

- 学习自己**被驳回的任务**，修正执行偏差

- 学习优秀历史案例，复刻高质量产出范式

- 学习高频bug、常见问题，提升一次通过率

- 学习工序标准化流程，减少流程失误

### 6\.3 学习状态（Learning）正式调度规则（可落地）

解决原有问题：学习状态无逻辑、无动作、空状态占位。

#### 6\.3\.1 学习触发时机（三种合法触发）

1. **被动触发（任务后复盘学习）**：Agent处理问题完成后，空闲状态下自动进入短时学习，复盘本次任务问题

2. **主动触发（空闲迭代学习）**：长时间无任务，自动切换learning状态，迭代岗位技能

3. **异常触发（纠错学习）**：问题处理失败、审核驳回、报错后，强制触发针对性纠错学习

#### 6\.3\.2 学习期间调度约束（核心规则）

- Agent处于**learning状态：不接单、不分配任务、不参与调度**

- 学习为**后台轻量迭代**，不阻塞系统主流程

- 高优先级任务到来：可中断学习 → 切为idle → 立即接单执行

- 学习完成自动落库记录，状态回归idle空闲

### 6\.4 学习内容落库设计（对接现有数据表）

无需新增表，复用现有结构，同时新增技能学习扩展字段逻辑，完全兼容旧架构：

1. **agent\_info表**：新增技能标签、学习进度、最近学习内容、通过率评分

2. **task\_flow\_log表**：新增「岗位技能学习」流转动作，记录每一次学习行为

3. **task\_exception\_log表**：失败任务自动关联对应技能，作为本次学习素材

### 6\.5 学习闭环逻辑（完整链路）

**任务处理 → 产出结果 → 老大审核（通过/驳回） → 记录优劣案例 → 空闲时针对性学习 → 更新自身技能模型 → 下次任务能力升级**

真正实现：**每个Agent各司其职、各学其长、越用越强**，不是统一大模型训练，是**岗位垂直持续进化**。
