### 2.2 API 端点文档模板

**文件**: `docs/04_接口文档/[module]-api.md`

```markdown
# [Module] API 端点文档

## GET /api/[module]/{id}
获取单个资源详情

### 请求头
```
Authorization: Bearer <token>
Content-Type: application/json
```

### 路径参数
| 参数 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| id | string | 是 | 资源 ID |

### 响应 (200 OK)
```json
{
  "success": true,
  "statusCode": 200,
  "data": {
    "id": "...",
    "name": "..."
  },
  "message": "Success",
  "timestamp": "2026-05-21T10:00:00Z"
}
```

### 错误响应 (404 Not Found)
```json
{
  "success": false,
  "statusCode": 404,
  "errors": null,
  "message": "Resource not found",
  "timestamp": "2026-05-21T10:00:00Z"
}
```
```

## 三、文档维护规范

### 3.1 文档命名规范
```
日期: YYYY-MM-DD 格式
阶段: PHASE_NN (e.g., PHASE_01, PHASE_07)
轮次: ROUND_NN (e.g., ROUND_001, ROUND_015)
模块: [module-name] (kebab-case)
版本: v[MAJOR].[MINOR].[PATCH]

示例:
- ROUND_REPORT_20260521_DEV_001.md
- PHASE_07_DELIVERY_SUMMARY.md
- AutoCodeForge-Architecture-v1.0.0-20260521.md
```

### 3.2 更新频率
| 文档类型 | 更新频率 | 负责人 |
|---------|---------|-------|
| MasterPlan | 每轮更新 | Strategic Planner |
| PROJECT_SPEC | 规范变更时 | @Auditor + Planner |
| 模块设计文档 | 模块完成时 | @Worker |
| API 文档 | 端点变更时 | @Worker |
| 代码审查报告 | 每轮完成时 | @Auditor |
| DAILY_REPORT | 每日晚间 | @Worker |
| README | 发版时 | @Worker |

### 3.3 文档质量检查清单
- [ ] 文档标题、版本、日期完整
- [ ] 内容结构清晰，标题分层正确
- [ ] 代码示例有 ✅/❌ 对比
- [ ] 表格格式规范，数据准确
- [ ] 链接有效，无断链
- [ ] 无拼写错误或不规范用词
- [ ] 技术术语准确（避免模糊表述）
- [ ] 已更新对应的目录与导航

### 3.4 文档权限与编辑规则
- **只读文档** (禁止手动编辑):
  - PROJECT_SPEC.md (仅 AI 更新)
  - MasterPlan.md (仅 AI 更新)
  
- **可编辑文档** (评审后更新):
  - 模块设计文档
  - API 文档
  - 开发规范
  
- **自动生成文档** (不手动编辑):
  - ROUND_REPORT
  - CODE_REVIEW_REPORT
  - DAILY_REPORT

## 五、文档演进与版本管理

### 5.1 文档版本策略
文档采用 Semantic Versioning：
- **MAJOR**: 架构重大调整或功能完全重做
- **MINOR**: 新增模块或功能
- **PATCH**: 缺陷修复、澄清说明、小的格式调整

### 5.2 文档备份与历史
所有发版文档保存在 `docs/reports/` 下，按日期和版本号归档：
```
docs/
├── AutoCodeForge-Architecture-v1.0.0-20260521.md  (当前版本)
├── AutoCodeForge-Architecture-v0.9.0-20260515.md  (历史版本)
└── ...
```

## 七、文档治理规范

### 7.1 文档冲突解决
如发现文档不一致（如 API 文档与代码不符）：
1. 优先级：**代码是单一真实来源**
2. 立即更新文档以匹配代码
3. 如存在多个版本，保留最新版本

### 7.2 文档废弃处理
过时文档标记为 `[DEPRECATED]` 并注明原因：
```markdown
# [DEPRECATED] 旧的 XXX 文档

**废弃原因**: 已由新的 XXX 文档替代  
**替代文档**: 参见 docs/02_架构设计/新文档.md  
**废弃日期**: 2026-05-21  

---
# 原内容...（保留以供参考）
```
