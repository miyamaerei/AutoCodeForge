# AutoCodeForge 代码合规校验清单

**文档版本**: v1.0  
**创建日期**: 2026-05-20  
**适用项目**: AutoCodeForge 后端  
**校验对象**: 所有后端代码、配置、文档

---

## 📋 执行概览

本文档定义了 AutoCodeForge 后端项目的全量代码合规校验标准，涵盖项目架构、代码质量、数据模型、API 设计、安全合规、性能要求、测试覆盖等多个维度。

### 校验等级说明

| 等级 | 标识 | 说明 | 必需性 |
|------|------|------|--------|
| **阻断性** | 🔴 | 违反将导致严重问题，必须修复 | **必选** |
| **重要** | 🟠 | 建议修复，否则可能影响系统质量 | **强烈推荐** |
| **一般** | 🟡 | 建议优化，提升代码可维护性 | **推荐** |
| **建议** | 🟢 | 最佳实践，可选执行 | **可选** |

---

## 🏗️ 1. 项目架构合规性检查

### 1.1 项目结构

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **四层架构** | 🔴 | 必须包含 Core、Infrastructure、Application、Api 四个项目 | 检查项目文件目录 |
| **Core 无依赖** | 🔴 | Core 项目不依赖 Infrastructure 或 Api，只依赖基础类库 | 检查项目引用 |
| **依赖方向** | 🔴 | Api → Application → Infrastructure → Core，不允许反向依赖 | 检查项目引用关系 |
| **命名空间** | 🟠 | 命名空间必须符合 `AutoCodeForge.{Layer}.{Module}` 规范 | 检查代码文件 |
| **目录结构** | 🟠 | 各层目录结构清晰，Entities、Repositories、Services、Endpoints、Middleware 分离 | 检查目录树 |

### 1.2 基础设施复用

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **基类实体** | 🔴 | 所有实体继承 AuditableEntity 或 UserOwnedEntity | 检查实体类定义 |
| **BaseRepository** | 🔴 | 所有 Repository 继承 BaseRepository<T> | 检查 Repository 类 |
| **统一响应** | 🔴 | 所有 API 响应使用 ApiResponse<T> | 检查 Endpoints 代码 |
| **异常中间件** | 🔴 | 全局异常处理中间件已注册并生效 | 检查 Program.cs |
| **依赖注入** | 🟠 | 所有服务通过 DI 注册，不使用 new 关键字实例化 | 检查代码 |

## 🔧 2. 代码质量标准与规范

### 2.1 编码规范

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **命名规范** | 🔴 | 类、接口、方法使用 PascalCase，参数、局部变量使用 camelCase | 检查代码 |
| **接口命名** | 🔴 | 接口必须以 I 开头（如 ICurrentUser） | 检查接口定义 |
| **函数注释** | 🔴 | 所有 public 方法必须有 XML 文档注释 | 检查代码注释 |
| **类型注释** | 🟠 | 所有 public 类、接口必须有 XML 文档注释 | 检查代码注释 |
| **空值处理** | 🟠 | 使用可空引用类型，避免 null 引用异常 | 检查代码 |
| **代码格式** | 🟡 | 使用 .editorconfig 统一代码格式，无多余空格、空行 | 检查代码格式 |
| **单行长度** | 🟢 | 单行代码不超过 120 字符 | 检查代码 |

### 2.2 代码质量

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **单一职责** | 🟠 | 每个类、方法只负责一个功能，职责清晰 | 代码审查 |
| **重复代码** | 🟠 | 避免重复代码，提取公共方法/类 | 代码审查 |
| **魔法数字** | 🟡 | 避免使用魔法数字，使用常量或枚举 | 检查代码 |
| **复杂度过高** | 🟡 | 单个方法圈复杂度不超过 10 | 使用代码分析工具 |
| **异常处理** | 🟠 | 只捕获能处理的异常，避免吞掉异常 | 检查异常处理代码 |

## 🗄️ 3. 数据模型合规性检查

### 3.1 实体设计

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **主键类型** | 🔴 | 所有实体主键必须是 Guid 类型 | 检查实体定义 |
| **基类继承** | 🔴 | UserEntity 继承 AuditableEntity，其他用户数据实体继承 UserOwnedEntity | 检查实体定义 |
| **软删除字段** | 🔴 | UserOwnedEntity 必须包含 IsDeleted 字段（基类已包含） | 检查实体定义 |
| **用户隔离字段** | 🔴 | UserOwnedEntity 必须包含 NtId 字段（基类已包含） | 检查实体定义 |
| **时间戳** | 🔴 | AuditableEntity 必须包含 CreatedAt、UpdatedAt 字段（基类已包含） | 检查实体定义 |
| **SugarTable 特性** | 🔴 | 所有实体必须添加 [SugarTable] 特性并指定表名 | 检查实体定义 |
| **SugarColumn 特性** | 🟠 | 主键必须添加 [SugarColumn(IsPrimaryKey = true)] | 检查实体定义 |
| **枚举定义** | 🟠 | 枚举类型与实体放在同一命名空间或单独的 Enums 目录 | 检查枚举位置 |

### 3.2 数据访问

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **Repository 继承** | 🔴 | 所有 Repository 继承 BaseRepository<T> | 检查 Repository 类 |
| **ICurrentUser 使用** | 🔴 | BaseRepository 通过 ICurrentUser 获取 NtId，不直接依赖 HttpContext | 检查 BaseRepository 代码 |
| **全局过滤器** | 🔴 | SqlSugar 全局过滤器自动过滤 IsDeleted=true 的记录 | 检查 SqlSugarSetup |
| **查询方法** | 🟠 | Repository 只提供数据访问方法，不包含业务逻辑 | 检查 Repository 代码 |
| **异步优先** | 🟠 | 所有数据访问方法使用 async/await | 检查代码 |

## 🌐 4. API 设计合规性检查

### 4.1 端点设计

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **路由规范** | 🔴 | API 路由使用 RESTful 风格，格式为 /api/v1/{resource}/{id} | 检查 Endpoints |
| **HTTP 方法** | 🔴 | GET（查询）、POST（创建）、PUT（更新）、DELETE（删除）使用正确 | 检查 Endpoints |
| **统一响应** | 🔴 | 所有 API 响应使用 ApiResponse<T> 格式 | 检查 Endpoints 代码 |
| **分页参数** | 🟠 | 列表接口使用 page、pageSize 参数，返回 PagedResult<T> | 检查列表接口 |
| **路径版本** | 🟠 | API 路径包含版本号（如 /api/v1/） | 检查路由 |

### 4.2 输入输出

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **DTO 分离** | 🟠 | 请求使用专门的 Request DTO，响应使用专门的 Response DTO，不直接暴露实体 | 检查代码 |
| **参数验证** | 🔴 | 请求参数必须进行验证（必填、格式、范围等） | 检查 DTO 验证特性 |
| **敏感数据** | 🔴 | 密码、API Key 等敏感信息不在响应中返回 | 检查响应数据 |
| **Swagger 文档** | 🟠 | 所有 API 端点有 XML 注释，Swagger 文档完整 | 检查 Swagger UI |

### 4.3 错误处理

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **HTTP 状态码** | 🔴 | 正确使用 HTTP 状态码（200、400、401、403、404、500） | 检查 Endpoints |
| **错误响应格式** | 🔴 | 异常响应使用 ApiErrorResponse 格式，包含 Code、Message、RequestId | 检查异常中间件 |
| **字段级错误** | 🟠 | 参数验证失败时返回字段级错误信息 | 检查验证响应 |

## 🔒 5. 安全合规性检查

### 5.1 认证授权

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **JWT 认证** | 🔴 | 所有需要认证的 API 使用 JWT Bearer Token | 检查 Program.cs 和 Endpoints |
| **Token 验证** | 🔴 | JWT 中间件验证 Token 有效性、过期时间 | 检查 JwtAuthMiddleware |
| **NtId 提取** | 🔴 | 从 JWT Token 中提取 NtId 并存储到 HttpContext 或 ICurrentUser | 检查认证流程 |
| **用户隔离** | 🔴 | 所有数据查询自动过滤 NtId，确保用户只能访问自己的数据 | 检查 BaseRepository |

### 5.2 数据安全

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **密码加密** | 🔴 | 密码使用 PBKDF2 或 Argon2 等安全算法加密，不存储明文 | 检查 PasswordHelper |
| **敏感数据加密** | 🔴 | API Key、凭据等敏感数据必须加密存储 | 检查 LLMModelConfig 等实体 |
| **软删除** | 🔴 | 删除操作使用软删除（IsDeleted=true），不物理删除数据 | 检查 Repository 删除方法 |
| **SQL 注入防护** | 🔴 | 使用 SqlSugar 原生查询方法，避免拼接 SQL | 检查数据访问代码 |

### 5.3 输入安全

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **输入验证** | 🔴 | 所有用户输入必须进行验证（长度、格式、范围） | 检查 DTO 验证 |
| **XSS 防护** | 🟠 | 输出用户输入的内容时进行适当转义 | 检查响应数据 |
| **路径遍历防护** | 🟠 | 文件操作时验证路径，避免路径遍历攻击 | 检查文件处理代码 |

## ⚡ 6. 性能要求检查

### 6.1 数据访问性能

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **异步查询** | 🔴 | 所有数据库查询使用 async/await | 检查 Repository 代码 |
| **分页查询** | 🟠 | 列表查询必须分页，避免一次性加载大量数据 | 检查列表接口 |
| **N+1 查询** | 🟠 | 避免 N+1 查询问题，使用 Includes 或 Join | 检查查询代码 |
| **索引优化** | 🟡 | 常用查询字段添加索引 | 检查实体定义、数据库配置 |

### 6.2 API 性能

| 检查项 | 等级 | 校验标准 | 验证方式 |
|--------|------|----------|----------|
| **响应时间** | 🟠 | API 响应时间不超过 500ms（不包括 AI 调用） | 性能测试 |
| **限流策略** | 🟠 | AI 调用等资源消耗大的接口实施限流 | 检查限流代码 |
| **超时配置** | 🟠 | LLM 调用、工具调用有合理的超时配置 | 检查超时设置 |

## 🔍 9. 代码审查清单

### 9.1 审查流程

| 检查项 | 等级 | 说明 |
|--------|------|------|
| **功能正确性** | 🔴 | 代码实现了预期功能，逻辑正确 |
| **代码质量** | 🟠 | 代码清晰、易读、易维护 |
| **安全考虑** | 🔴 | 没有安全漏洞，符合安全规范 |
| **性能影响** | 🟠 | 考虑了性能影响，没有明显的性能问题 |
| **测试覆盖** | 🟠 | 相关功能有测试覆盖 |
| **文档更新** | 🟡 | 更新了相关文档（如需要） |

### 9.2 常见问题检查

| 检查项 | 等级 | 说明 |
|--------|------|------|
| **硬编码字符串** | 🟡 | 避免硬编码，使用常量或配置 |
| **TODO 注释** | 🟡 | 及时处理 TODO 注释，或创建 Issue 跟踪 |
| **注释与代码不一致** | 🟠 | 注释与代码保持一致 |

## ✅ 10. 验收标准

### 10.1 阶段验收

| 阶段 | 必须通过的检查项 | 验收标准 |
|------|----------------|----------|
| **阶段一（基础设施）** | 所有 🔴 级架构和基础设施检查项 | 项目可以编译运行，基础架构符合规范 |
| **阶段二（数据层）** | 所有 🔴 级实体和数据访问检查项 | 数据库可以自动创建，认证功能正常 |
| **后续阶段** | 新增功能相关的 🔴 和 🟠 级检查项 | 新功能符合规范，现有功能不受影响 |

### 10.2 发布验收

发布前必须满足以下条件：
- 所有 🔴 级检查项通过
- 至少 80% 的 🟠 级检查项通过
- 无已知的安全漏洞
- 核心功能测试通过
- 性能满足基本要求
